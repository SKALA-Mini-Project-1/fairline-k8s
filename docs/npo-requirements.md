# Fairline NPO Requirements

## 1. 문서 목적

이 문서는 Fairline MSA 플랫폼의 비기능 요구사항(NPO, Non-Functional Objectives / Requirements)을 정의한다.
기능 요구사항이 "무엇을 할 것인가"를 설명한다면, NPO는 "어떤 품질 수준으로 운영할 것인가"를 설명한다.

본 문서는 현재 저장소의 실제 운영 구조를 기준으로 작성되었다.

- Kubernetes 기반 EKS 배포
- Spring Boot MSA 구성
- Redis 사용
- PostgreSQL RDS 사용
- 향후 Outbox / CDC / Kafka / Debezium 확장 예정

---

## 2. 시스템 범위

본 NPO는 아래 서비스와 공통 인프라에 적용한다.

- `user-auth-service`
- `concert-service`
- `ticketing-service`
- `payment-service`
- `queue-service`
- `gateway`
- `frontend`
- `redis`
- `postgresql RDS`

---

## 3. NPO 설계 원칙

Fairline의 비기능 요구사항은 다음 원칙을 따른다.

1. 예약, 결제, 좌석 상태와 같은 핵심 데이터는 성능보다 일관성을 우선한다.
2. 단기 데모/개발 편의보다 운영 안정성과 장애 복구 가능성을 우선한다.
3. 현재는 단일 RDS를 공유하되, PostgreSQL schema 분리를 통해 서비스 경계를 유지한다.
4. 향후 서비스 간 직접 DB 의존을 줄이기 위해 Outbox / CDC 기반 구조로 확장 가능해야 한다.
5. 장애가 발생했을 때 원인 추적이 가능하도록 로그, 헬스체크, 배포 이력을 확보해야 한다.

---

## 4. 핵심 비기능 요구사항

### 4.1 가용성

시스템은 일반적인 배포, 단일 Pod 장애, 일시적 네트워크 오류 상황에서도 주요 사용자 기능을 계속 제공할 수 있어야 한다.

요구사항:

- `gateway`, `frontend`는 롤링 배포 중에도 외부 진입점이 유지되어야 한다.
- 핵심 백엔드 서비스는 단일 Pod 재시작 시 자동 복구되어야 한다.
- readiness probe와 liveness probe를 통해 비정상 Pod가 트래픽을 계속 받지 않아야 한다.
- 장애 복구 후 수동 개입 없이 서비스가 `Running` 및 `Ready` 상태로 회복되어야 한다.

운영 메모:

- 현재 `concert-service`는 RDS connection slot 이슈를 고려하여 일시적으로 `1 replica` 운영이 허용된다.
- 향후 RDS 스펙 상향 또는 연결 풀 최적화 이후 멀티 replica 운영을 재검토한다.

성공 기준:

- 단일 서비스 Pod 재시작 시 Kubernetes가 자동으로 새 Pod를 기동한다.
- 배포 후 `kubectl rollout status` 기준으로 정상 완료되어야 한다.

### 4.2 데이터 일관성

예약, 결제, 좌석 상태, 팬스코어와 같은 핵심 도메인 데이터는 서로 충돌하거나 이중 처리되면 안 된다.

요구사항:

- `ticketing-service`의 예약 생성, 예약 만료, 좌석 선점/해제는 중복 처리에 안전해야 한다.
- `payment-service`의 결제 승인, 실패, 환불은 중복 호출 및 재시도 상황에서도 일관된 결과를 유지해야 한다.
- 좌석 상태와 예약 상태는 비즈니스 규칙에 따라 전이되어야 하며, 임의 상태 점프가 발생하면 안 된다.
- 사용자 팬스코어는 최소한 음수 누적, 중복 가산, 교차 서비스 동기화 실패에 대해 방어되어야 한다.
- 향후 이벤트 기반 구조로 전환하더라도, 원천 데이터 변경과 이벤트 기록은 논리적으로 동일한 업무 단위로 보장되어야 한다.

성공 기준:

- 동일 결제/예약 요청의 재시도 시 데이터 중복 생성이 발생하지 않는다.
- 예약 확정 후 좌석이 중복 판매되지 않는다.

### 4.3 성능

시스템은 일반적인 사용자 사용 시 체감 지연이 크지 않도록 응답 속도를 유지해야 한다.

요구사항:

- 로그인, 회원 정보 조회, 콘서트 목록 조회, 대기열 상태 조회는 빠른 응답을 제공해야 한다.
- 예약/결제 API는 데이터 정합성을 우선하되, 과도한 지연이 발생하지 않아야 한다.
- Redis를 사용하는 대기열 및 잠금 관련 기능은 DB보다 우선적으로 메모리 기반 처리 경로를 사용해야 한다.
- DB connection pool은 RDS 용량을 초과하지 않는 범위에서 운영되어야 한다.

권장 목표:

- 콘서트 조회/대기열 상태 조회: 일반 상황에서 수 초 이내 응답
- 로그인/회원 조회: 일반 상황에서 수 초 이내 응답
- 예약/결제 처리: 정합성을 유지하면서도 사용자가 실패로 오해하지 않을 수준의 응답 시간 유지

운영 메모:

- 현재 RDS는 소형 인스턴스이므로 Hikari pool 크기를 보수적으로 유지한다.
- 성능 튜닝은 replica 확대보다 connection, query, Redis hit ratio 점검을 먼저 수행한다.

### 4.4 확장성

시스템은 향후 트래픽 증가와 기능 확장에 대응할 수 있어야 한다.

요구사항:

- 서비스는 schema 기반으로 도메인 경계를 유지해야 한다.
- 향후 `outbox_event` 테이블을 추가하여 이벤트 기반 구조로 확장 가능해야 한다.
- Kafka 및 Debezium 기반 CDC를 붙일 수 있도록 DB 설정과 서비스 구조가 이를 방해하지 않아야 한다.
- 사용자 조회/대기열/팬스코어/결제 후속 처리 등은 장기적으로 비동기화 가능한 형태로 발전할 수 있어야 한다.

성공 기준:

- 서비스 간 직접 DB 의존을 점진적으로 줄일 수 있는 구조를 유지한다.
- RDS logical replication, schema ownership, connector 추가가 가능해야 한다.

### 4.5 보안

시스템은 인증 정보, 사용자 정보, 결제 관련 정보를 안전하게 다뤄야 한다.

요구사항:

- DB 계정, JWT secret, 메일 계정, 결제 연동 키 등은 안전하게 관리되어야 한다.
- 운영 환경에서 비밀값은 평문 파일 커밋보다 Secret 또는 외부 비밀 저장소 기반 관리가 우선되어야 한다.
- 서비스 간 호출은 최소 권한 원칙을 따르는 방향으로 관리되어야 한다.
- 사용자 인증 정보와 결제 관련 민감 정보는 로그에 평문으로 남지 않아야 한다.

운영 메모:

- 현재 `secret.yaml` 기반 운영은 임시 운영 방식으로 간주한다.
- 장기적으로는 ExternalSecret, SealedSecret 또는 클라우드 시크릿 매니저 연동이 바람직하다.

### 4.6 장애 복구 및 복원력

시스템은 부분 장애가 전체 장애로 번지지 않도록 설계되어야 한다.

요구사항:

- 개별 서비스 실패 시 다른 서비스가 연쇄적으로 모두 중단되지 않아야 한다.
- Redis 장애, RDS 일시 오류, Pod 재시작 등에서 서비스가 자동 재기동 가능해야 한다.
- 타임아웃, 재시도, 상태 만료, 스케줄러 보정 로직을 통해 운영 중 잔여 데이터 불일치를 최소화해야 한다.
- 장애 발생 시 최소한 `kubectl logs`, `kubectl describe`, rollout 상태만으로도 1차 진단이 가능해야 한다.

성공 기준:

- 단일 Pod CrashLoop 시 원인 로그를 확보할 수 있어야 한다.
- 배포 실패 시 이전 Pod가 완전히 사라지기 전에 새 Pod readiness를 통해 방어할 수 있어야 한다.

### 4.7 운영성

운영자는 배포, 점검, 롤백, 장애 대응을 일관된 방식으로 수행할 수 있어야 한다.

요구사항:

- 매니페스트 구조는 서비스별로 분리되어 있어야 하며 변경 영향 범위를 예측할 수 있어야 한다.
- 운영 변경 사항은 문서화되어야 한다.
- RDS 전환, schema 분리, connection pool 제한 같은 인프라성 변경은 추적 가능해야 한다.
- 서비스별 환경 변수와 의존 인프라는 명시적으로 확인 가능해야 한다.

성공 기준:

- 특정 서비스의 DB 연결 방식, Secret 사용 여부, 배포 상태를 짧은 명령으로 확인할 수 있다.
- 주요 인프라 변경 이력이 문서로 남아 있다.

### 4.8 관측성

시스템은 성능 저하와 장애를 빠르게 감지할 수 있어야 한다.

요구사항:

- 애플리케이션 시작, DB 연결, 스케줄러 동작, 오류 발생 등 핵심 운영 이벤트가 로그에 남아야 한다.
- readiness/liveness probe를 통해 비정상 인스턴스를 식별할 수 있어야 한다.
- RDS connection exhaustion, Redis connection failure, rollout failure와 같은 주요 운영 장애는 로그 기반으로 빠르게 식별 가능해야 한다.

향후 강화 항목:

- 중앙 로그 수집
- 메트릭 수집 및 대시보드
- 알람 기준 정의

### 4.9 유지보수성

시스템은 팀 단위 협업과 이후 확장을 고려하여 수정 가능해야 한다.

요구사항:

- 서비스별 책임 경계가 문서화되어야 한다.
- 공통 인프라 변경과 서비스별 변경을 분리해서 추적할 수 있어야 한다.
- 대규모 구조 변경 전에는 schema, deployment, connector, secret 변경 지점을 예측 가능해야 한다.

성공 기준:

- 신규 팀원이 문서와 매니페스트만 보고도 현재 배포 구조를 이해할 수 있다.
- 운영 중 민감 파일과 생성 산출물이 버전 관리에 무분별하게 포함되지 않아야 한다.

## 5. 현재 구조에서의 특별 고려사항

### 5.1 Shared RDS + Schema 분리

현재는 완전한 database-per-service가 아니라 shared RDS + schema-per-service 모델을 채택한다.

따라서 다음을 추가로 고려해야 한다.

- schema search path 설정이 잘못되면 서비스가 `public` 또는 타 schema를 참조할 수 있다.
- 일부 서비스는 현재 구조상 여러 schema를 조회하므로, 완전한 독립 서비스로 보기 어렵다.
- 향후 Outbox / CDC 구조 도입 시 cross-schema 직접 조회 의존은 점진적으로 줄이는 것이 바람직하다.

### 5.2 RDS Connection Capacity

현재 RDS 인스턴스는 작은 개발용 스펙이므로 connection pool과 replica 수를 과도하게 키우면 배포 중 connection slot 부족이 발생할 수 있다.

따라서 다음 정책을 적용한다.

- 기본 Hikari pool은 보수적으로 운영한다.
- replica 확대 전 DB connection 사용량을 먼저 점검한다.
- 롤링 배포 시 old Pod와 new Pod가 잠시 공존하는 상황을 고려한다.

### 5.3 Outbox / CDC 확장 준비

향후 이벤트 기반 비동기 구조를 도입할 때의 목표는 다음과 같다.

- 서비스 간 직접 DB 조회를 줄인다.
- 핵심 비즈니스 변경과 이벤트 발행의 정합성을 높인다.
- Kafka 기반 비동기 소비 모델을 붙일 수 있도록 준비한다.
- Debezium은 business table 전체보다 `outbox_event` CDC 중심으로 설계하는 것이 바람직하다.

## 6. 우선순위

현재 Fairline에서 가장 중요한 NPO 우선순위는 다음과 같다.

1. 데이터 일관성
2. 가용성
3. 장애 복구 가능성
4. 보안
5. 운영성
6. 성능
7. 확장성
8. 관측성 고도화

우선순위를 이렇게 둔 이유는, 본 시스템이 단순 조회 서비스가 아니라 예약/결제/좌석 상태를 다루는 트랜잭션 중심 시스템이기 때문이다.

## 7. 수용 기준 요약

본 문서는 다음 조건을 만족하면 최소 수용 가능 상태로 본다.

- 모든 핵심 백엔드 서비스가 EKS에서 정상 배포 및 자동 복구 가능해야 한다.
- 서비스는 RDS를 사용하며, schema 기반 도메인 경계를 유지해야 한다.
- 예약/결제/좌석 상태에 대해 중복 처리 및 불일치 방지 로직이 있어야 한다.
- Secret, DB 연결, 롤링 배포, 장애 로그 확인이 운영 절차로 가능해야 한다.
- 향후 Outbox / CDC / Kafka 구조를 붙일 수 있도록 구조적 확장 가능성을 유지해야 한다.

## 9. 현재 저장소 기준 실행 체크리스트 (2026-05-01 재점검)

아래 항목은 `fairline-k8s` 저장소를 다시 점검한 결과, 현재 상태와 후속 작업 우선순위를 함께 정리한 것이다.

### 9.1 이미 반영된 항목

- RDS shared instance + schema search path 구조가 매니페스트에 반영되어 있다.
- 서비스별 Hikari pool 상한이 낮게 설정되어 있어 소형 RDS connection slot 리스크를 완화하고 있다.
- 모든 Deployment에 GPU 노드 제외 affinity가 적용되어 있다.
- 모든 핵심 Deployment에 readiness / liveness probe가 정의되어 있다.
- RDS logical replication 활성화 및 schema 분리 작업 이력이 `docs/rds-worklog.md`에 문서화되어 있다.

### 9.2 우선 처리 권장 항목

- `secret.yaml`에 실제 DB 계정과 JWT secret이 평문으로 존재하므로, 외부 비밀 저장소 또는 암호화된 Secret 관리 방식으로 전환해야 한다.
- 일부 서비스가 여러 schema를 search path에 포함하고 있으므로, cross-schema 직접 조회를 줄이는 구조 개선이 필요하다.
- 현재 `infra/postgres`가 여전히 저장소에 남아 있으므로, RDS 전환 이후 in-cluster postgres의 역할을 명확히 하거나 제거 기준을 문서화해야 한다.
- 관측성은 probe와 로그 확인 수준에 머물러 있으므로 중앙 로그, 메트릭, 알람 체계가 추가로 필요하다.

### 9.3 다음 작업 순서 제안

1. Secret 관리 방식을 `secret.yaml` 직접 커밋에서 ExternalSecret, SealedSecret, 또는 AWS Secrets Manager 연동으로 전환한다.
2. 서비스별 readiness 기준을 단순 tcpSocket에서 애플리케이션 health endpoint 기반으로 고도화한다.
3. cross-schema 조회 지점을 식별하고 Outbox / CDC 전환 후보 흐름을 선정한다.
4. in-cluster postgres 제거 여부와 제거 전 검증 절차를 운영 문서로 고정한다.
